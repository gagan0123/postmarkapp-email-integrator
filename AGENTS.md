# Agents Guide ‚Äî Postmarkapp Email Integrator

## Project Overview

A WordPress plugin that overrides `wp_mail()` to send emails through the Postmark API. This is a community fork of the official Postmark plugin, fixing critical bugs and adding features like Cc/Bcc/Reply-To support, settings import, and open tracking.

- **Slug**: `postmarkapp-email-integrator`
- **Text Domain**: `postmarkapp-email-integrator`
- **Minimum WordPress**: 5.0
- **License**: GPLv2 or later

## File Structure


## Architecture

The plugin is a single-file architecture (`postmarkapp.php`). Key sections:

1. **Activation & Settings Import** ‚Äî On activation, imports settings from the official Postmark plugin if no existing config is found
2. **Admin UI** ‚Äî Settings page under Settings > Postmarkapp with fields for API key, sender address, force HTML, and track opens
3. **wp_mail Override** ‚Äî When enabled, conditionally defines `wp_mail()` to route emails through Postmark's API
4. **Header Parsing** ‚Äî `pma_parse_headers()` handles string and array headers, extracting Content-Type, Cc, Bcc, Reply-To
5. **API Communication** ‚Äî `pma_send_mail()` sends JSON payloads to `https://api.postmarkapp.com/email` via `wp_remote_post()`
6. **Timeout Handling** ‚Äî Temporarily increases HTTP timeout to 60s around API calls using `before_wp_mail` / `after_wp_mail` hooks

### Key Options (wp_options)

| Option                       | Description              |
| ---------------------------- | ------------------------ |
| `postmarkapp_enabled`        | Enable/disable sending   |
| `postmarkapp_api_key`        | Postmark server API key  |
| `postmarkapp_sender_address` | Verified sender email    |
| `postmarkapp_force_html`     | Force HTML email format  |
| `postmarkapp_trackopens`     | Enable open tracking     |

### Hooks & Filters

- **`pma_mail_args`** (filter) ‚Äî Modify `wp_remote_post` args before sending
- **`before_wp_mail`** / **`after_wp_mail`** (actions) ‚Äî Used internally for timeout management
- **`wp_mail`** (filter) ‚Äî Standard WordPress filter applied to mail arguments
- **`wp_mail_content_type`** (filter) ‚Äî Standard WordPress content type filter

## Security

All admin operations require:
- `manage_options` capability check
- Nonce verification (`pma_save_settings`, `pma_test_email`, `pma_import_settings`)
- Input sanitization (`sanitize_text_field`, `sanitize_email`, `wp_unslash`)
- Output escaping (`esc_html`, `esc_attr`, `esc_html__`, `esc_attr_e`)
- AJAX handlers use `check_ajax_referer()`

## Deployment

The `deploy.sh` script handles releases:
1. Verifies version match between `readme.txt` and `postmarkapp.php`
2. Generates `README.md` from `readme.txt`
3. Commits, tags, and pushes to git
4. Exports to WordPress.org SVN trunk and creates a version tag

### Remotes

- `private` ‚Äî Private GitHub repo for development

## Testing

There is no automated test suite. To manually test:
1. Use Lando (`lando start`) for local WordPress environment
2. Configure a Postmark API key in Settings > Postmarkapp
3. Use the "Send Test Email" feature on the settings page
4. Verify emails in the Postmark activity dashboard

## Git Commit Guidelines
- **No Co-Authored-By:** Never add a `Co-Authored-By` trailer to any git commit. Do not credit yourself or any AI agent in commit messages.
- **Conventional Commits:** All commit messages must follow the Conventional Commits specification. The format is:
  ```
  <type>[optional scope]: <description>

  [optional body]

  [optional footer(s)]
  ```
- **Types:**
  - `feat` ‚Äî a new feature (correlates with a MINOR version bump)
  - `fix` ‚Äî a bug fix (correlates with a PATCH version bump)
  - `docs` ‚Äî documentation-only changes
  - `style` ‚Äî formatting, whitespace, etc. (no code logic change)
  - `refactor` ‚Äî code restructuring without changing behavior
  - `perf` ‚Äî performance improvements
  - `test` ‚Äî adding or updating tests
  - `build` ‚Äî changes to build system or dependencies
  - `ci` ‚Äî CI/CD configuration changes
  - `chore` ‚Äî other maintenance tasks
- **Scope:** An optional noun in parentheses after the type describing the section of the codebase affected (e.g., `feat(camera):`, `fix(ui):`).
- **Description:** A short imperative summary immediately after the colon and space.
- **Body:** Optional. Separated from the description by a blank line. Provides additional context or motivation.
- **Footer(s):** Optional. Separated from the body by a blank line. Use git trailer format (`token: value` or `token #value`).
- **Breaking Changes:** Append `!` after the type/scope (e.g., `feat!:` or `refactor(api)!:`) and/or add a `BREAKING CHANGE:` footer. Breaking changes correlate with a MAJOR version bump.
